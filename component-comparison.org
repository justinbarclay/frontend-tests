* Overview

All three repos implement the same two conceptual components against an identical store
interface (=useBuilderStore=) and =FieldSchema= type:

- =ConfigurationForm= — a form for editing a field's schema (label, name, placeholder, type,
  and type-specific validation/config).
- =PreviewField= — a live read-only simulation of what the configured field would look like to
  an end user.

#+ATTR_LATEX: :environment tabularx :width \linewidth :align lXXX
| Repo           | UI library                         | Lines: ConfigurationForm | Lines: PreviewField |
|----------------+------------------------------------+--------------------------+---------------------|
| jolly-ui       | React Aria / Jolly UI              |                      173 |                  91 |
| shadcn-baseui  | shadcn-ui primitives + BaseUI      |                      188 |                 113 |
| mantine-v8     | Mantine v8                         |                      173 |                  72 |

Mantine achieves the most with the fewest lines in =PreviewField= (72) because its component
API accepts all behaviour through props rather than composing primitives manually.

* ConfigurationForm

** Shared structure

All three follow the same pattern:

1. Render shared fields: =label=, =name=, =placeholder=, field =type= selector.
2. Dispatch to a type-specific sub-configurator (=TextConfigurator=, =NumberConfigurator=,
   =SelectConfigurator=, or an empty =<div>= for boolean).
3. Write back to the store via =updateRoot=, =updateValidation=, and =updateConfig=.

** Layout

#+ATTR_LATEX: :environment tabularx :width \linewidth :align lXXX
| Aspect             | jolly-ui                          | shadcn-baseui                    | mantine-v8                     |
|--------------------+-----------------------------------+----------------------------------+--------------------------------|
| Grid               | 2-column, fields use col-span-2/4 | 12-column, fields use col-span-6 | 12-column, fields use span={6} |
| Grouping primitive | Plain =<div>=                     | =<FieldGroup>= wrapper component | =<Grid>= / =<Grid.Col>=        |
| Label/Name row     | Stacked (each col-span-2)         | Side-by-side (col-span-6 each)   | Side-by-side (span={6} each)   |

** UI library components

#+ATTR_LATEX: :environment tabularx :width \linewidth :align lXXX
| Field          | jolly-ui           | shadcn-baseui                       | mantine-v8                             |
|----------------+--------------------+-------------------------------------+----------------------------------------|
| Text input     | =JollyTextField=   | =<Field>= + =<Input>=               | =<TextInput>=                          |
| Number input   | =JollyNumberField= | =<Field>= + =<Input type="number">= | =<NumberInput>=                        |
| Select         | =JollySelect=      | Shadcn =<Select>= + Trigger/Content | =<Select>=                             |
| Add-option btn | =<div>= flex wrapper + =<Button onPress>= | =<Field>= → =<InputGroup>= → =<InputGroupAddon>= → =<InputGroupButton>= | =rightSection={<Button>}= on TextInput |

Both =jolly-ui= and =shadcn-baseui= required compositional setup before reaching the button itself.
=jolly-ui= wraps =JollyTextField= and =<Button>= in a plain =<div className="flex items-end gap-2">=
— no library primitive involved. =shadcn-baseui= requires a four-level component stack: =<Field>=
provides label/error context; =<InputGroup>= establishes the combined input frame; =<InputGroupAddon>=
positions the trailing slot; and finally =<InputGroupButton>= renders the clickable control. Mantine
sidesteps this entirely by accepting a =rightSection= prop directly on =<TextInput>=.

** SelectItem identity

- =jolly-ui= passes =id= to =<SelectItem id="number">= (React Aria convention).
- =shadcn-baseui= passes =value= to =<SelectItem value="text">= (Radix/shadcn convention).
- =mantine-v8= passes a plain =data={["number", "select", "text", "boolean"]}= string array
  directly to =<Select>=; no child item components are needed.

** SelectConfigurator — option list rendering

- =jolly-ui= renders each existing option with its own numbered label (/"Option 1"/, /"Option 2"/ …).
- =shadcn-baseui= groups all existing options under a single /"Options"/ =FieldLabel=, then renders
  bare =<Input>= elements beneath it.
- =mantine-v8= mirrors =jolly-ui=: each option gets its own =<TextInput label="Option N">= and adds
  a new-option field with the =+= button embedded in the input's =rightSection=.

* PreviewField

** Shared structure

All three export a default =Preview= component that reads =state.field= from the store, looks up
the appropriate sub-renderer in a dispatch map, and renders it. The empty/unsupported-type
fallback messages are identical across all three.

** NumberPreview

** SelectPreview

- =jolly-ui= uses =JollySelect= with =<SelectItem id={option}>= children.
- =shadcn-baseui= uses the full Radix Select stack (=Trigger= / =Content= / =Item=) with
  =defaultValue= set to the placeholder and =<SelectItem value={option}>=.
- =mantine-v8= uses a single =<Select label={field.label} data={field.config.options}>=;
  no child components are needed.

** Select Examples
I think looking at the Select Examples side-by-side is the most illustrative of the differences in approach between the three libraries, so here are the relevant snippets from each repo's =ConfigurationForm=.
*** shadcn
shadcn's Select element is the most primitive because the library provides no higher-level abstractions for form fields — you compose the full trigger/content/item stack yourself and wire it up to the store manually. This gives you maximum control but also means more [[id:4c832f7a-cf68-4e37-a81d-b6bd299fee9c][boilerplate]] and manual wiring. Which means that this is something we'll have to explicitly write within out component/design system.

*** jollyui
jolly-ui provides a slight higher-level =JollySelect= component that abstracts away the trigger/content/item structure and internal wiring, so you can just pass =onChange= and it gives you back the selected value. However, it still requires you to render =SelectItem= children and pass the item ID.

*** mantine
Finally, Mantine's =<Select>= is the most high-level: it abstracts away the trigger/content/item structure like jolly-ui, but it also eliminates the need for child item components entirely by accepting a plain array of strings via the =data= prop. This is the most concise and straightforward approach, but it also means you have less control over the rendering of individual items (e.g. you can't easily add icons or custom layouts to the options without switching to a more manual approach).
[[https://github.com/justinbarclay/frontend-tests/blob/main/mantine-v8/src/components/ConfigurationForm.tsx#L160][Source: ConfigurationForm.tsx]]

      data={["number", "select", "text", "boolean"]}

    />
  </Grid.Col>
#+end_src
** Number Input Example
Here, the differences are less pronounced because all three libraries provide a number input component that handles the core functionality. However, shadcn's approach still requires more manual wiring and boilerplate: you have to render a =<Field>= wrapper, an =<Input type="number">=, and then manually implement the decimal formatting and error hint logic. In contrast, jolly-ui's =JollyNumberField= provides declarative formatting via props and built-in error handling, while Mantine's =<NumberInput>= also handles formatting internally (though it doesn't display an error state for empty values). So while the core input functionality is similar across all three, shadcn's approach is more low-level and requires more manual work to achieve the same features.

** Error handling Example
Finally, I am not going to beat a dead horse anymore, but you can see the clear pattern in the 3 systems. Jolly UI provides the most built-in error handling and declarative validation options, shadcn provides the most manual control but requires you to implement all error handling and validation logic yourself, and Mantine falls somewhere in between with some built-in validation but less explicit error state display (no error messages, just red highlights). So again, shadcn gives you maximum flexibility at the cost of more boilerplate and manual work, while jolly-ui and Mantine provide more out-of-the-box functionality with less control over the details.

** Custom Components and Styling
When it comes to creating custom components and styling Jolly UI and shadcn become a lot more closely aligned. Both libraries require you to compose lower-level primitives together to achieve custom layouts and styles, which gives you a lot of flexibility but also means more boilerplate, more verbosity, and manual work. Mantine, on the other hand, provides more high-level components with built-in styling and layout options, which can make it faster to build common UI patterns. But then when you need to implement CSS modules.

I am sure that this section of the demo is not _entirely_ representative of what it would look like in production because we would have made wrapper components to handle some of the generic styling and layout concerns, but I think it still serves to illustrate the general differences in approach between the three libraries when it comes to building custom components and applying styles.

** jolly-ui violations

*** ConfigurationForm — select type
- *Rule*: =button-name= (critical)
- *Description*: Ensure buttons have discernible text.
- *Element*: The =JollySelect= trigger button rendered by React Aria inside =SelectConfigurator=
  (the select used to add options) renders with no visible label text in jsdom — the button is
  present but its accessible name is empty.
- *Root cause*: React Aria's =<Select>= wraps the trigger in a =<button>= whose text content comes
  from the selected item label; with no item selected and no explicit =aria-label=, the button is
  nameless.

*** PreviewField — boolean field
- *Rule*: =label= (critical)
- *Description*: Ensure every form element has a label.
- *Element*: =<input data-react-aria-pressable="true" role="switch" type="checkbox">=
- *Root cause*: =jolly-ui='s =BooleanPreview= renders the label text in a sibling =<span>= before
  the switch but does not associate it via =htmlFor= / =aria-labelledby=. React Aria's =<Switch>=
  requires the label to be passed as a child or via its =<Label>= sub-component to wire the
  association automatically.

** shadcn-baseui violations

*** ConfigurationForm — all four types (text, number, select, boolean)
- *Rule*: =button-name= (critical)
- *Description*: Ensure buttons have discernible text.
- *Element*:
  #+begin_src html
  <button type="button" role="combobox" aria-expanded="false" aria-haspopup="listbox"
          data-slot="select-trigger" …>
  #+end_src
- *Root cause*: The field =type= selector at the top of =ConfigurationForm= is a BaseUI/shadcn
  =<Select>= whose =<SelectTrigger>= is a bare =<button>=. The sibling =<FieldLabel>= that visually
  labels it is not programmatically associated with the trigger (BaseUI's lower-level =SelectTrigger=
  does not inherit from the surrounding =<Field>= context the way Mantine or React Aria do). This
  affects every ConfigurationForm variant because the type selector is always rendered.

*** PreviewField — select field
- *Rule*: =button-name= (critical)
- *Description*: Same root cause as above; the =<SelectTrigger>= in =SelectPreview= also has no
  accessible name.

*** PreviewField — boolean field
- *Rule*: =aria-toggle-field-name= (serious)
- *Description*: Ensure every ARIA toggle field has an accessible name.
- *Element*: =<span role="switch" aria-checked="false" data-slot="switch" …>=
- *Root cause*: BaseUI's =<Switch>= renders as a =<span role="switch">= without wiring to its
  sibling =<FieldLabel>=. An explicit =aria-label= or =aria-labelledby= on the switch element is
  required.

** mantine-v8 — no violations

All 8 tests pass. Mantine's component API wires accessibility automatically:

- =<Select label="…">= passes the label string through to the underlying combobox trigger via
  =aria-label= / =aria-labelledby=, so =button-name= is never violated.
- =<Switch label="…">= associates the label with the toggle via its internal =<label>= element,
  satisfying both =label= and =aria-toggle-field-name=.

** Mind–body dualism: Boilerplate and Granularity
:PROPERTIES:
:ID:       4c832f7a-cf68-4e37-a81d-b6bd299fee9c
:END:

The distinction between "Boilerplate" and "Component Granularity" is often a matter of architectural lens rather than technical difference. When a library requires a multi-level compositional stack to render a standard form field—as seen in the primitive-first approach of *Shadcn* or the headless orchestration required by *Jolly UI (React Aria)* —it is correctly identified as *Boilerplate* in the context of Enterprise or SaaS applications where speed of delivery and system-wide consistency are the primary value drivers. In these environments, every line of "glue code" represents a maintenance tax that adds no unique business value. However, that same verbosity is viewed as **Granularity** when a product’s core differentiator relies on high-precision interaction design—such as surgical re-render optimizations or non-standard brand aesthetics—that a "black box" monolithic component like **Mantine** would otherwise obstruct. Ultimately, the choice is a decision on where to place the "Extensibility Insurance": most applications are better served by the guardrails of a high-level API, while only a small subset of "hero features" truly justify the overhead of managing the low-level primitives and state-wiring inherent in Shadcn and Jolly UI.
